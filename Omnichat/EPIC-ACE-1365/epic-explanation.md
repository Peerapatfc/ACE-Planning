# EPIC ACE-1365: Contact Profile and Commerce Context — คำอธิบายแบบเข้าใจง่าย

## สรุปภาพรวม
Epic นี้คือการสร้าง "โปรไฟล์ลูกค้าแบบรวมศูนย์" (Contact Profile) เพื่อให้ Agent เห็นภาพรวมของลูกค้าคนเดียวกันที่ติดต่อมาผ่านหลายช่องทาง (เช่น ทัก LINE แล้วไปทัก Facebook ต่อ) โดยจะรู้ได้ว่าเป็นคนเดียวกัน และดูประวัติทั้งหมดรวมถึงโน้ตระดับติดต่อ (Contact Note) ได้ในที่เดียว

**เปรียบเทียบง่าย ๆ:**
เหมือนมี "แฟ้มประวัติคนไข้" ที่ไม่ว่าคนไข้จะไปรักษาแผนกไหน (ช่องทางไหน) หมอก็จะเห็นประวัติเก่า ข้อมูลตัวตน และสมุดโน้ตประจำตัวของคนไข้คนนั้นครบถ้วนในแฟ้มเดียว

---

## 1. แนวคิดหลัก (Core Concepts)

### 1.1 ความแตกต่างระหว่าง Identity กับ Contact
- **Identity (ตัวตนในแต่ละช่องทาง):** ข้อมูลลูกค้าที่ผูกกับช่องทาง เช่น LINE_ID, Facebook_PSID, TikTok_ID
- **Contact (ผู้ติดต่อรวม):** ตัวตนระดับ Omnichat (Contact ID) ที่ใช้รวม Identity ของช่องทางต่าง ๆ เข้าด้วยกัน ว่าเป็น "คนเดียวกัน"

### 1.2 กฎการรวมข้อมูล (Merge Rules v1)
1. **รวมอัตโนมัติเฉพาะช่องทางเดียวกัน (Deterministic Auto Link):** ถ้าเป็น LINE user_id เดียวกัน จะรวมเป็น Contact เดียวกันเสมอ
2. **ขอบเขตระดับร้านค้า (Marketplace Shop Scoping):** ช่องทาง Marketplace (TikTok/Shopee/Lazada) จะรวมกันเฉพาะภายใน shop_id เดียวกันเท่านั้น ไม่ข้ามร้าน
3. **ห้ามรวมข้ามช่องทางอัตโนมัติ (No Cross-Channel Auto-Merge):** แม้เบอร์โทรหรืออีเมลจะตรงกัน ก็จะไม่รวมช่องทาง LINE กับ FB หรือ Shopee ให้อัตโนมัติใน v1 (จะใช้ระบบ "แนะนำให้รวม" แทน)
4. **รวมเองโดย Agent (Manual Link):** Agent สามารถกดรวมช่องทางเข้าด้วยกัน (Link) ได้เอง พร้อมระบบเก็บประวัติ (Audit log) และย้อนกลับได้ (Undo)

## 2. ส่วนประกอบบนหน้าจอ (UI Components)

### Contact Profile Panel (แผงข้อมูลติดต่อ)
เมื่อแอดมินเปิดห้องแชท แผงด้านข้างจะแสดง:
- **Overview:** รูป, ชื่อที่ใช้แสดง, ป้ายกำกับ (Tags), เวลาที่เห็นล่าสุด, จำนวนแชททั้งหมด
- **Identities:** รายชื่อช่องทางทั้งหมดที่คนคนนี้เคยใช้ติดต่อมา (เช่น มีโลโก้ LINE พ่วงด้วยโลโก้ Facebook)
- **Key Events:** ไทม์ไลน์เหตุการณ์สำคัญล่าสุด 5 รายการ (เช่น เปลี่ยนสถานะการแชท, SLA, การเพิ่มโน้ต)
- **Recent Conversations:** ประวัติแชทล่าสุด กดแล้วนำทางไปดูแชทเก่าได้

### Customer Notes (สมุดจดประจำตัวลูกค้า)
- เป็นโน้ตที่ผูกติดกับ **Contact (ลูกค้า)** ไม่ใช่ระดับ Conversation ภายในแชทนั้นๆ
- พิมพ์ครั้งเดียว แอดมินทุกคนในบริษัท (Tenant) เห็นโน้ตนี้เหมือนกัน ไม่ว่าจะเปิดแชทลูกค้ารายนี้จากช่องทางไหนก็ตาม
- สามารถ **ปักหมุด (Pin)** โน้ตสำคัญไว้บนสุดได้ (เช่น เป็นลูกค้า VIP หรือ Blacklist)
- มี **Audit Trail** บันทึกเสมอว่า ใครเป็นคนเขียน/แก้ไข และทำเมื่อไหร่

## 3. ลำดับการทำงาน (User Flow)

#### สถานการณ์: ลูกค้าชื่อ "สมชาย" ทักมาจาก LINE และหลังจากนั้นทักมาทาง Facebook

1. **ครั้งแรก - ทักผ่าน LINE:**
   - ระบบยังไม่เคยรู้จักสมชาย จึงสร้าง Identity 1 (LINE_ID) และสร้าง Contact ใหม่ (Contact_A)
   - Agent คุยปกติ และทิ้ง Customer Note ไว้ว่า "ลูกค้าชอบสีแดง"
2. **ครั้งที่สอง - ทักผ่าน Facebook:**
   - สมชายทักมาทาง Facebook ระบบสร้าง Identity 2 (FB_PSID) และสร้าง Contact ใหม่ (Contact_B)
   - ระบบตรวจพบว่า อีเมลหรือเบอร์ที่สมชายให้ไว้ในระบบ ตรงกันทั้งใน LINE และ FB
   - ระบบ **ไม่รวมอัตโนมัติ** (No Auto-Merge v1) แต่แสดงข้อความแจ้งเตือน Agent (Suggestion) ว่า "ข้อมูลนี้อาจเป็นคนเดียวกัน แนะนำให้เชื่อมโยงข้อมูล"
3. **การรวมตัวตน (Manual Link):**
   - Agent กด "เชื่อมโยง (Link)"
   - ระบบจะนำ Identity 2 เข้าไปผูกอยู่ใต้ Contact_A
   - ตอนนี้ Contact_A มีทั้งตัวตนใน LINE และ FB
4. **ผลลัพธ์:**
   - เมื่อ Agent เปิดแชท Facebook ของคุณสมชาย จะเห็น Customer Note "ลูกค้าชอบสีแดง" ที่เคยพิมพ์ไว้ตอนคุยไลน์ทันที
   - ในแผง Contact Profile จะเห็นว่า "คุณสมชาย" มี 2 Identities คือ LINE และ Facebook

## สรุปรายละเอียด (Scope v1)

| หมวดหมู่ | รายละเอียด |
|---|---|
| Stories ที่พัฒนาใน Phase นี้ | 3 Stories (Contact Profile, Merge Rules, Customer Notes) |
| Stories ที่อยู่ใน Backlog | 3 Stories (Last Purchase, Order Details, Order Actions) |
| กฎการรวมอัตโนมัติ (Auto-Merge) | เฉพาะใน Identity แบบเดียวกัน/Shop เดิมเท่านั้น ห้ามรวมข้าม Channel อัตโนมัติ |
| การรวมข้อมูลข้าม Channel | ใช้ Manual Link โดย Agent เท่านั้น (มีระบบ Audit log + กด Undo ได้) |
| Customer Note | ระดับ Contact (ลูกค้ารายบุคคล), ปักหมุดได้, ทุก Agent เห็นตรงกัน |

## 4. คำอธิบายแต่ละ Story (Story Breakdown)

### STORY-CTX-01: Contact Profile (แผงข้อมูลติดต่อลูกค้า)
**เป้าหมาย:** สร้างแผงข้อมูลด้านข้าง (Side Panel) เพื่อให้ Agent เห็นภาพรวมของลูกค้าได้ทันทีโดยไม่ต้องเปลี่ยนหน้าจอ
**สิ่งที่จะทำ:**
- **Overview:** แสดงรูปโปรไฟล์, ชื่อ, ป้ายกำกับ (Tags), เวลาใช้งานล่าสุด, ช่องทางติดต่อหลัก (เช่น เบอร์โทร/อีเมล)
- **Identities:** แสดงตราสัญลักษณ์ (Badge) ของทุกช่องทางที่ลูกค้ารายนี้ใช้ (เช่น LINE, Facebook)
- **Key Events:** แสดงเหตุการณ์สำคัญล่าสุด 5 รายการ เช่น การเปลี่ยนสถานะแชท, SLA, หรือมีการล็อกโน้ต
- **Recent Conversations:** แสดงประวัติการแชทเก่า ๆ ของลูกค้ารายนี้ทั้งหมด สามารถกดเพื่อกลับไปดูแชทเก่าได้

### STORY-CTX-02: Identity Model and Merge Rules v1 (กฎการรวมบัญชีลูกค้า)
**เป้าหมาย:** ป้องกันปัญหารวมข้อมูลผิดคน (Data Merge Conflict) ด้วยกฎกติกาที่ชัดเจนในเวอร์ชันแรก
**สิ่งที่จะทำ:**
- **การรวมอัตโนมัติ (Auto-Merge):** ทำได้เฉพาะเมื่อเป็นบัญชีจากแพลตฟอร์มเดียวกันเป๊ะ ๆ เท่านั้น (เช่น LINE ID เดิม) และถ้าเป็นแพลตฟอร์มขายของ (Marketplace) จะต้องอยู่ในร้าน (Shop) เดียวกันด้วย
- **การแนะนำ (Suggestion):** หากระบบพบว่ามีอีเมลหรือเบอร์โทรตรงกันข้ามแพลตฟอร์ม (เช่น LINE กับ Facebook) ระบบจะ **ไม่รวมให้อัตโนมัติ** แต่จะแค่ "แนะนำ (Suggest)" ให้ Agent ทราบ
- **การจัดการโดย Agent (Manual Action):** Agent สามารถกดรวม (Link) หรือ แยกร่าง (Unlink) บัญชีได้ด้วยตัวเอง โดยทุกการกระทำจะถูกบันทึก (Audit Log) และหากกดผิด สามารถกด Undo ได้

### STORY-CTX-03: Customer Notes (สมุดจดประจำตัวลูกค้า)
**เป้าหมาย:** ให้ Agent สามารถจดบันทึกข้อมูลสำคัญของลูกค้าที่ทุกคนในทีมต้องรับรู้ร่วมกัน 
**สิ่งที่จะทำ:**
- **จดที่เดียว เห็นทุกคน:** เป็นโน้ตที่ผูกกับตัวบุคคล (Contact) ไม่ใช่ผูกกับห้องแชท ดังนั้นไม่ว่าลูกค้าจะทักมาช่องทางไหน หรือคุยกับ Agent คนใด ก็จะเห็นโน้ตนี้
- **ระบบปักหมุด (Pin):** สามารถปักหมุดข้อความสำคัญไว้บนสุด (เช่น "ลูกค้าด่าเก่ง ระวัง", "ลูกค้า VIP")
- **ความโปร่งใส (Audit Trail):** บันทึกเสมอว่า Agent คนไหนเป็นคนเพิ่มหรือแก้ไขโน้ตนี้ และทำเมื่อเวลาใด
