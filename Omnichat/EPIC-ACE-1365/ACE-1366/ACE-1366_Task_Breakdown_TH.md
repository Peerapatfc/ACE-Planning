# EPIC-ACE-1365 / ACE-1366 รายละเอียดงาน (Task Breakdown)
**Story**: STORY-CTX-01_Contact_Profile (หน้าต่าง Profile + เหตุการณ์สำคัญ + ประวัติ)

## 1. งานฝั่ง Backend / API
### 1.1 สร้าง API สำหรับดึงข้อมูล Contact Context
- **Task**: เตรียม API Endpoint `GET /api/v1/contacts/:contact_id/context`
- **Description**: สร้าง API endpoint ใหม่เพื่อดึงข้อมูล detail ทั้งหมดของ contact นำมาแสดงบน profile panel ด้านขวา
- **ข้อมูลที่ส่งคืน (Endpoint Data)**:
  - `overview`: ข้อมูลทั่วไปของลูกค้า (ชื่อ, รูปโปรไฟล์, วันที่สร้าง/ออนไลน์ล่าสุด) ดึงจากโมเดล `Contact`
  - `identities`: บัญชีการติดต่อจากแพลตฟอร์มต่างๆ ดึงจาก `ChannelAccount`
  - `key_events`: รายการเหตุการณ์สำคัญ 5 รายการล่าสุด (เช่น การปรับสถานะ SLA หรือผลจากการทำงานของ `ConversationAssignment`)
  - `recent_conversations`: ประวัติการสนทนาล่าสุด 5-10 รายการเรียงตาม `last_message_at DESC` ดึงจาก `Conversation`
  - `last_purchase_stub`: ข้อมูลจำลองสำหรับประวัติการซื้อ (เตรียมไว้รองรับการพัฒนาต่อ)
  - `customer_notes`: ข้อมูลจำลองสำหรับบันทึก Note (เตรียมไว้รองรับการพัฒนาต่อ)

### 1.2 จัดการฐานข้อมูลสำหรับการดึงประวัติ
- **Task**: ปรับแต่ง Prisma Queries (Data Layer Queries for Context)
- **Description**: ทำการเขียน Prisma query สำหรับรวบรวม Identities, Event Feeds, และ ประวัติการแชท ให้สามารถส่งข้อมูลไปที่ Frontend ได้ในคราวเดียว เพื่อเพิ่มความรวดเร็ว

## 2. งานฝั่งฐานข้อมูล (Database Tasks)
### 2.1 ปรับแต่ง Schema
- **Task**: ตรวจสอบแหล่งข้อมูลสำหรับดึง SLA Event
- **Description**: ตรวจสอบความถูกต้องของตาราง หาก SLA event จำเป็นต้องใช้ field ใหม่ ให้ใส่เตรียมไว้ใน `ConversationAssignment` , `Message` หรือ `Conversation` 

## 3. งานฝั่งหน้าบ้าน (Frontend UI Tasks)
### 3.2 สร้างชิ้นส่วน Profile Panel
- **Task**: สร้างคอมโพเนนต์ `ContactProfilePanel`
- **Description**: จัดทำหน้าต่างฝั่งขวาที่พับเก็บได้ (Collapsible) สำหรับแสดงรายละเอียด Contact โดยจัดเรียงตามที่ทีมออกแบบ (Overview -> Identities -> ประวัติการสั่งซื้อล่าสุด -> โน้ตลูกค้า -> เหตุการณ์สำคัญ -> ประวัติการสนทนา)

### 3.3 ไทม์ไลน์แจ้งเตือนเหตุการณ์ขนาดเล็ก
- **Task**: สร้างรายการ Key Events
- **Description**: เพิ่มฟีดเหตุการณ์สำคัญขนาดย่อมลงใน Panel แสดงเหตุการณ์ล่าสุด 5 รายการ พร้อมแจ้งคนทำและเวลา

### 3.4 ระบุช่องทางการสนทนา และการแก้ปัญหาตอนที่ข้อมูลหาย
- **Task**: จัดการรูปแบบป้ายกำกับ (Badges) และตัวโหลดหน้าจอ (Placeholders)
- **Description**: จัดการ UI ไม่ให้พังในกรณีที่มีข้อมูลเข้ามาไม่ครบ และใช้งาน Skeleton โหลดหน้าจอรอระหว่างเรียก API พร้อมแสดงสัญลักษณ์แพลตฟอร์มการติดต่อให้ครบถ้วน

### 3.5 เชื่อมโยงประวัติไปยังห้องแชท
- **Task**: การย้ายหน้าไปยัง Inbox
- **Description**: ทำให้เวลาที่ผู้ใช้คลิกประวัติแชทล่าสุด จะย้ายไปยังห้องแชทนั้นได้ทันทีโดยที่ข้อความ Draft ฝั่งหน้า Inbox ปัจจุบันไม่หายไป

## 4. งานทดสอบซอฟต์แวร์ (QA & Testing)
### 4.1 ชุดทดสอบ (Test Cases)
- **TC-01 (ทดสอบ Identities)**: จำลองให้ลูกค้ามีการสนทนาจากหลายช่องทาง -> หน้าจอด้านขวาต้องแสดงป้ายของทุกช่องทางถูกต้อง 
- **TC-02 (ทดสอบ Key Events)**: จำลองให้เกิดเหตุการณ์เปลี่ยนผู้ดูแล (Assigning) จำนวนหลายครั้ง -> แถบเหตุการณ์ต้องแสงผลเหตุการณ์ 5 ครั้งล่าสุดได้ตามเวลาจริง
- **TC-03 (กดประวัติเปลี่ยนแชท)**: ลูกค้าเคยสนทนาไว้ 3 ห้อง -> ฝั่งขวากดไปที่ห้องอื่น -> ระบบต้องเปลี่ยนเป็นห้องนั้นใน Inbox ทันที
- **TC-04 (สถานการณ์ข้อมูลมาไม่ครบ)**: ตัดข้อมูลบางส่วนออกจากระบบ API -> ระบบฝั่งขวาต้องแสดง Placeholder/สัญลักษณ์ว่าง โดยที่หน้าเว็บไม่เละ
- **TC-05 (ด้านประสิทธิภาพ)**: ระหว่างรอผล API สามารถแสดงผล Skeleton โหลดข้อมูลได้ราบรื่น และไม่หน่วงหน้าแชทหลัก
