# Subtask 3: Implement Rate Limiting & Backoff Retries

## รายละเอียด
พัฒนาระบบป้องกันการเรียก API ถี่เกินไป (Rate Limits) และใช้กลไกหน่วงเวลา (Exponential Backoff) เมื่อเจอ Error ชั่วคราว เพื่อรักษาสุขภาพของระบบเราและป้องกันไม่ให้ Vendor แบนเราจากการยิงขยะรัวๆ.

## รายละเอียดการพัฒนา
1. **ไฟล์ที่เกี่ยวข้อง**: 
   - `apps/omnichat-service/src/polling/services/polling-worker.service.ts`
   - ไฟล์เกี่ยวกับ Retry Policy ของ Worker คิว.
2. **สถานะปัจจุบัน**: ไม่มี (เป็นระบบสร้างใหม่ทั้งหมด).
3. **สิ่งที่ต้องทำ**: 
   - บังคับใช้และตรวจสอบ Rate Limit ของ API ยี่ห้อต่างๆ (TikTok, Shopee, Lazada) ไว้ตั้งแต่ฝั่งเรา เพื่อให้ Worker ไม่ยิงรีเควสแบบปูพรมใส่ Server เขา
   - ถ้าระบบปลายทางตอบกลับด้วย `429 Too Many Requests` หรือ `5xx Server Error` ให้ถือว่าเป็น Transient Failure (ล่มชั่วคราว)
   - กรณีนี้ ห้ามทิ้ง (Drop) งาน แต่ให้ส่ง `PollJob` กลับเข้าคิวและหน่วงเวลาการประมวลผลให้ช้าลงเรื่อยๆ แบบรอบทวีคูณ (Exponential backoff)
   - หากแพลตฟอร์มใดแพลตฟอร์มหนึ่งพังติดๆกันนานมาก ควรมี Circuit Breaker ตัดสัญญาณหยุดพักคิวของแพลตฟอร์มนั้นทิ้งพักนึง เพื่อให้ Worker ของเราสามารถเอาเวลาไปรันคิวแพลตฟอร์มอื่นที่ยังไม่ล่มได้
