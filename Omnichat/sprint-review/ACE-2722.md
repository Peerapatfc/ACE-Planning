# Scenario รีวิว (demo)

> Demo script พร้อมใช้: `scripts/test-rbac.sh ADMIN_EMAIL SUPERVISOR_EMAIL AGENT_EMAIL`
> หลักการเช็ค: role ที่ไม่มีสิทธิ์ → คาดหวัง 403 (middleware บล็อกก่อนถึง handler), role ที่มีสิทธิ์ → ไม่ใช่ 403 (400/404 จาก id ปลอมถือว่าผ่าน gate แล้ว)

## ตารางสิทธิ์ที่ต้องรีวิว

| Permission | Admin | Supervisor | Agent | ตัวอย่าง endpoint |
|---|:--:|:--:|:--:|---|
| ไม่ต้องมี permission (แค่เป็นสมาชิก) | ✓ | ✓ | ✓ | ดู inbox, ตอบแชท, เปลี่ยน status, ดู broadcast/automation |
| `org:conversation:assign` | ✓ | ✓ | ✗ | `PATCH /conversations/:id/assign` |
| `org:sla:manage` | ✓ | ✓ | ✗ | `PUT /workspaces/sla-config` |
| `org:notification:manage` | ✓ | ✓ | ✗ | `PUT /workspaces/notification-rules` |
| `org:automation:manage` | ✓ | ✓ | ✗ | สร้าง/แก้/toggle automation rule |
| `org:broadcast:manage` | ✓ | ✓ | ✗ | สร้าง/แก้/ลบ/ยกเลิก broadcast |
| `org:contact_tag:manage` | ✓ | ✓ | ✗ | rename contact tag |
| `org:channel:manage` | ✓ | ✗ | ✗ | connect/แก้/ลบ channel account |
| `org:member:manage` | ✓ | ✗ | ✗ | จัดการสมาชิก, invite, เปลี่ยน role |
| `org:workspace:manage` | ✓ | ✗ | ✗ | แก้ workspace, business hours, **ลบ** contact tag / automation rule |

## Scenario 1 — สิทธิ์ระดับ admin เท่านั้น
- **ขั้นตอน:** ยิง `POST /channel-accounts` และ `PATCH /workspaces` ด้วย token ของ agent, supervisor, admin
- **คาดหวัง:** agent → 403, supervisor → 403, admin → ผ่าน gate

## Scenario 2 — สิทธิ์ admin + supervisor (agent ไม่ได้)
- **ขั้นตอน:** ยิง `PATCH /conversations/:id/assign`, `PUT /workspaces/sla-config`, `POST /automation-rules` ด้วยทั้ง 3 role
- **คาดหวัง:** agent → 403, supervisor → ผ่าน, admin → ผ่าน

## Scenario 3 — "ลบ" เข้มกว่า "สร้าง/แก้" (delete = admin only)
- **ขั้นตอน:** supervisor ยิง `DELETE /automation-rules/:id` และ `DELETE /contact-tags/:id`
- **คาดหวัง:** supervisor → 403 (delete ต้อง `org:workspace:manage`), admin → ผ่าน
- **จุดโชว์:** supervisor สร้าง/แก้ automation & contact tag ได้ แต่ลบไม่ได้

## Scenario 4 — action ที่ทุก role ทำได้ต้องไม่โดนบล็อก
- **ขั้นตอน:** agent ยิง `GET /conversations`, `PATCH /conversations/:id/status`, `POST /contacts/:id/tags/:tagId` (ผูก tag เข้า contact), `GET /automation-rules`
- **คาดหวัง:** ผ่านหมด (ไม่ 401/403)

## Scenario 5 — ไม่มี JWT / JWT ไม่ถูกต้อง
- **ขั้นตอน:** ยิง endpoint ใดก็ได้แบบไม่มี `Authorization` header
- **คาดหวัง:** 401 Unauthorized (ClerkAuth บล็อกก่อน RBAC)

## Scenario 6 — ABAC / tenant isolation
- **ขั้นตอน:** ใช้ token ของ workspace A ยิงอ่าน/แก้ resource ของ workspace B
- **คาดหวัง:** 404 หรือผลลัพธ์ว่าง — **ห้าม 500 และห้ามเห็นข้อมูลข้าม tenant**

## Scenario 7 — route สาธารณะต้องไม่โดนบล็อก
- **ขั้นตอน:** เรียก widget endpoint (`/widget/*`), webhook, OAuth callback โดยไม่มี JWT
- **คาดหวัง:** ทำงานปกติ — middleware RBAC ครอบเฉพาะ `/v1` ที่ต้อง auth, ไม่กระทบ route สาธารณะ

## Scenario 8 — เปลี่ยน role แล้ว token เก่ายังถือสิทธิ์เดิม (พฤติกรรมที่ต้องเล่าให้ทีมรู้)
- **ขั้นตอน:** admin ลด role ของ user จาก supervisor → agent ขณะ user นั้นยัง active อยู่
- **คาดหวัง:** token เก่ายังใช้สิทธิ์ supervisor ได้จนกว่า JWT จะ refresh (Clerk session token ~60 วินาที) เพราะ permission ฝังอยู่ใน JWT claim
- **จุดสำคัญ:** ไม่ใช่ bug — เป็น trade-off ของ JWT-based permission ควรแจ้งทีมให้เข้าใจ window นี้
