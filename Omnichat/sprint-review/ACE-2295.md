# Scenario รีวิว (demo)

> Demo script พร้อมใช้: `scripts/test-broadcast-audience.sh`
> Endpoint: `GET /broadcasts/audience/tags`, `GET /broadcasts/audience/estimate`, `POST/PATCH /broadcasts`
> ค่าจริงในโค้ด: `targetingType` = `all | specific_people | specific_chat`; tag `mode` = `include | exclude` (per tag)

## Scenario 1 — โหมด "ส่งทุกคน" (all)
- **ขั้นตอน:** เลือกโหมด all
- **คาดหวัง:** ซ่อนช่องเลือก tag, badge แสดง 100%, estimate = จำนวน contact ทั้งหมดของ LINE OA นั้น

## Scenario 2 — โหมด "เจาะจงคน" (specific_people)
- **ขั้นตอน:** เลือกโหมด specific_people → เปิด dropdown tag
- **คาดหวัง:** dropdown ค้นหาได้, เรียง A-Z, แสดงจำนวน contact ต่อ tag; เลือก tag → ขึ้น chip + estimate อัปเดตทันที; ลบ chip → estimate อัปเดตกลับ

## Scenario 3 — โหมด "เจาะจงแชท" (specific_chat)
- **ขั้นตอน:** เลือกโหมด specific_chat → เลือก tag จาก chat library
- **คาดหวัง:** ใช้ chat tag library (คนละชุดกับ people) — **ห้ามปน people-tag กับ chat-tag ใน broadcast เดียว**

## Scenario 4 — validation ต้องเลือกอย่างน้อย 1 tag
- **ขั้นตอน:** อยู่โหมด specific แต่ไม่เลือก tag แล้วกดต่อ
- **คาดหวัง:** ขอบแดง + ข้อความ error ภาษาไทย, ไปต่อไม่ได้

## Scenario 5 — estimate dedup ข้ามหลาย tag + budget 2 วินาที
- **ขั้นตอน:** เลือก tag ที่ contact ทับกัน 2-3 อัน
- **คาดหวัง:** นับ contact ไม่ซ้ำ (COUNT DISTINCT), ตอบกลับ < 2 วินาที, ตัวเลข = จำนวนที่จะส่งจริง

## Scenario 6 — estimate ผูกกับ LINE OA ที่เลือก (จาก BC-01)
- **ขั้นตอน:** สลับ LINE OA
- **คาดหวัง:** estimate นับเฉพาะ contact ที่มี `line_user_id` ของ OA ที่เลือก

## Scenario 7 — ขอบเขตของ estimate
- **ขั้นตอน:** ตรวจกรณี conversation ที่ถูกลบ และโหมด specific ที่ไม่ได้เลือก tag
- **คาดหวัง:** ไม่นับ conversation ที่ถูกลบ; specific แต่ไม่มี tag → estimate = 0

## Scenario 8 — cap 500 คนตอนส่ง (specific modes)
- **ขั้นตอน:** เลือก audience specific ที่ estimate เกิน 500 แล้วกดส่ง
- **คาดหวัง:** จำกัดที่ 500 recipients (โหมด all ไม่จำกัด)

## Scenario 9 — RBAC
- **ขั้นตอน:** สร้าง/แก้ broadcast ด้วย agent / supervisor / admin
- **คาดหวัง:** ดู (`GET /broadcasts`) = ทุก role; สร้าง/แก้/ลบ/ยกเลิก = admin + supervisor (`org:broadcast:manage`), agent → 403

## Scenario 10 — draft ผ่อนปรน แต่ send เข้ม (มี unit test ยืนยัน)
- **ขั้นตอน:** เซฟ draft โหมด specific โดยไม่เลือก tag → สำเร็จ; เซฟ draft ที่ audience เกิน 500 → สำเร็จ; แล้วลองกดส่ง/schedule ทั้ง 2 กรณี
- **คาดหวัง:** draft เซฟได้ทั้งคู่ แต่ send/schedule โดน block (ไม่มี tag → reject, เกิน 500 → reject)
- **จุดสำคัญ:** เล่าในรีวิวว่าเป็น design ตั้งใจ — ไม่ใช่ bug ที่ draft เซฟเกิน 500 ได้

## Scenario 11 — boundary 500 พอดี (มี unit test ยืนยัน)
- **ขั้นตอน:** ส่ง broadcast ที่ audience = 500 พอดี และอีกอันที่ 501
- **คาดหวัง:** 500 พอดี = ผ่าน, เกิน = reject

## Scenario 12 — กัน state ขัดแย้งจาก client (มี unit test ยืนยัน)
- **ขั้นตอน:** ยิง API ตรง: (a) โหมด all แต่แนบ targetTags มาด้วย, (b) tag เดียวกันเป็นทั้ง include และ exclude, (c) mode มั่ว เช่น "banana"
- **คาดหวัง:** reject ทั้ง 3 กรณีด้วย validation error — tag_type ถูก derive ฝั่ง server จากโหมด ส่ง people-tag เข้าโหมด chat ไม่ได้โดย construction

## Scenario 13 — ลบ tag จาก library แล้ว estimate อัปเดตเอง (SQL กรอง deleted_at แล้ว)
- **ขั้นตอน:** สร้าง draft ที่เลือก tag X ไว้ → admin ลบ tag X จาก library → เปิด draft ดู estimate
- **คาดหวัง:** estimate ไม่นับ tag ที่ถูกลบ (ลดลง/เป็น 0) ไม่ error

## Scenario 14 — estimate นับ contact ที่ merge แล้วผ่านกลุ่ม (ทำเสร็จ 2026-07-17, E2E ยืนยัน)
- **ขั้นตอน:** tag Z อยู่บน contact B → merge A←B (ทั้งคู่มี LINE identity บน OA เดียวกัน) → estimate ด้วย tag Z
- **คาดหวัง:** recipients นับทั้ง A และ B (tag เป็น group-shared) — ก่อน merge = 1, หลัง merge = 2, unmerge แล้วกลับเป็น 1; จำนวนใน dropdown (audience tags count) ขยับตามด้วย
- **จุดโชว์:** ตัวเลข estimate ตรงกับที่ panel แสดง tag เสมอ — ไม่มีช่องว่างที่ agent เห็น tag แต่ broadcast ไม่นับ

## ⚠️ จุดที่ต้องแจ้งในรีวิว — ClickUp/Figma ไม่ตรงกับที่ทำจริง
- **ทำจริง:** 3 โหมด (all / specific_people / specific_chat), tag library ไม่ merge, รองรับ include/exclude per tag ใน schema
- **ClickUp AC1.1 + Figma:** ยังเป็นดีไซน์เก่า (dropdown รวม, 2 โหมด)
- **exclude mode:** ถูกตัดออกวันเดียวกัน (2026-07-14) — schema ยังรองรับถ้าจะรื้อกลับมา ไม่ต้อง migrate
- **ท่าที่แนะนำ:** เสนอว่าทำเสร็จตามมติทีม 2026-07-13 แล้วขอ PO เซ็นอัปเดต ClickUp AC หลังรีวิว
