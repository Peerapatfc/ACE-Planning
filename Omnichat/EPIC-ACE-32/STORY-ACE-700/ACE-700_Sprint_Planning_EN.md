## 2. Recommendations & Edge Cases to Watch Out For

1. **The 24-Hour Policy Window**:
   - **Risk**: Instagram strictly enforces a standard 24-hour messaging window for human agents to reply. Sending texts outside this window will throw an API error.
   - **Recommendation**: Error handling in Subtask 3 must distinctively identify the "outside 24-hour window" error code so the frontend can relay to the admin why a message failed.

2. **Attachment Handling**:
   - **Risk**: Users frequently share Reels or Story mentions to DMs. These show up as attachments or unsupported shares in the webhook.
   - **Recommendation**: Ensure the inbound mapper safely drops unsupported payload components or strings them out as `[Share/Reel Unsupported in v1]` without throwing an unhandled exception that blocks the queue.

3. **Instagram Threading Model**:
   - **Risk**: `MessagesService` creates threads based on `tenant_id`, `channel_account_id` and `external_user_id`. 
   - **Recommendation**: This is architecturally sound for IG v1. The developer just needs to guarantee `sourceUserId` mapped in the ingress payload matches the `id` property from Instagram's `sender` object.
